# KB Iteration Agent

The KB iteration agent is a deterministic maintenance workflow for a LightRAG knowledge base. It helps an operator inspect the current graph, summarize it for later LLM review, calculate quality signals, prepare reviewable improvement proposals, and compare iteration runs.

It is not a medical fact editor. It does not automatically mutate extracted medical facts, prompts, ontology rules, hierarchy rules, relation rules, WebUI behavior, or workspace data. LLM output is treated as analysis and proposal material, not as source of truth.

## Safety Boundary

The first deterministic runner is intentionally read-only with respect to the LightRAG workspace. It reads graph and input files, then writes reports under an iteration output directory.

The workflow may automatically:

- Generate graph snapshots and statistics.
- Generate compact Markdown memory for LLM and human review.
- Run deterministic quality checks.
- Write quality reports and score JSON.
- Write proposal queues and backlog files when proposals are supplied.
- Compare two snapshots and write a diff report.

The workflow must not automatically:

- Add medical facts that are not grounded in imported source documents.
- Correct fact-level KG data without explicit approval.
- Edit extraction prompts, ontology, aliases, hierarchy rules, relation rules, or WebUI display behavior.
- Delete, clear, rebuild, or rename a workspace.
- Treat LLM conclusions as evidence.

Generated files can help an LLM understand the state of the KB, but source-grounded evidence is still required before any medical or KG mutation is accepted.

## Deterministic Modules

The current deterministic implementation lives in `lightrag/kb_iteration/`:

- `snapshot.py`: builds `KGSnapshot` objects from GraphML and writes snapshot JSON through `write_snapshot_artifacts`.
- `markdown.py`: writes `kb_context.md`, `entity_catalog.md`, `relation_catalog.md`, `kg_structure.md`, and initializes memory files through `write_markdown_memory`.
- `quality.py`: evaluates deterministic quality metrics through `evaluate_snapshot_quality` and writes `quality_report.md` plus `snapshots/quality_score.json` through `write_quality_artifacts`.
- `medical_schema.py`: provides the built-in hospital-facing Medical Relationship Schema v1 used by medical LLM review prompts and deterministic legacy-relation migration checks.
- `proposals.py`: validates structured `ImprovementProposal` records and writes `approval_queue.md` and `improvement_backlog.md`.
- `diff.py`: compares snapshots through `compare_snapshots` and writes `diff_report.md` plus `snapshots/diff_summary.json` through `write_diff_report`.
- `runner.py`: orchestrates the first deterministic run through `run_iteration`.

`run_iteration` validates the workspace name before joining paths, so unsafe names such as path traversal attempts are rejected before reports are written.

The runner currently writes the core snapshot, Markdown, quality, and iteration log artifacts. It does not create proposal objects, proposal YAML files, or populated queue entries. Diff artifacts are written by `diff.py` when before/after snapshots are available.

## First Deterministic Run

Use the first run after ingestion or after a workspace rebuild to create a review package.

1. Confirm the LightRAG workspace already has `graph_chunk_entity_relation.graphml`.
2. Run `run_iteration` with the workspace, storage root, input root, output root, and optional profile.
3. Read `kb_context.md` first for a compact overview.
4. Review `quality_report.md` and `snapshots/quality_score.json`.
5. Review `approval_queue.md`, `improvement_backlog.md`, `known_issues.md`, and `quality_rules.md` as persistent memory files.
6. Stop at the recorded `pending_user_review` phase unless a human explicitly approves a mutation or rebuild.

Example:

```python
from lightrag.kb_iteration.runner import run_iteration

result = run_iteration(
    workspace="influenza_medical_v1",
    storage_root="data/rag_storage",
    input_root="data/inputs",
    output_root="work/kb-iteration",
    profile="clinical_guideline_zh",
)

print(result.output_dir)
print(result.quality_score.overall)
```

The first runner appends `phase: pending_user_review` to `iteration_log.md`. That phase means the review package is ready, not that any proposed change has been generated or applied.

## LLM Review Loop

After deterministic artifacts exist and an `LLMReviewClient` is configured or injected, maintainers can run the optional LLM review loop. The loop reads `snapshots/kg_snapshot.json`, `snapshots/quality_score.json`, `accepted_changes.md`, and `rejected_changes.md`. It builds focused context packages under `review_context/`, writes `llm_review_trace.json`, writes `llm_review_report.md`, writes `proposals.generated.yaml`, and updates the approval queue through the existing proposal validator.

The LLM review loop does not apply patches, mutate KG facts, edit prompts or rules, or rebuild a workspace. LLM output is analysis and proposal material only. Mutation proposals still require approval.

## Current Runtime Baseline

As of 2026-06-24, the recommended Agent runtime for the medical KG workflow is:

- model: `deepseek-v4-flash`
- mode: `agent_pipeline`
- default `max_proposals_per_run`: `200`
- mutation gate: human approval followed by deterministic allowlisted apply

The latest real `influenza_medical_v1` review cycle is fully audited. The initial medical review saw `53` proposals: `3` were accepted, `50` were rejected, and the 3 accepted changes were deterministically applied to the KG with `0` blocked changes. A first rerun generated `15` proposals and all 15 were rejected. A second rerun regenerated the same 15 semantic candidates with `-2` proposal ID suffixes; all 15 were rejected again.

The current cleaned runtime queue has no pending human-review proposals. The rejected records remain in `rejected_changes.md`, apply evidence remains in `accepted_changes_apply_result.md`, and archived run artifacts remain under `codex_runs/`. This proved the P1 high-yield funnel can produce queueable proposals, but also exposed the next bottleneck: exact proposal-ID memory is insufficient. The next source fix should suppress equivalent candidates by `semantic_fingerprint` and `execution_fingerprint`, not by proposal ID alone.

The proposal orchestrator now collects both `quality_score.details.medical_schema_issues` and `quality_score.details.entity_cleanup_issues` into a raw issue ledger before task packing. A raw issue is one detected defect from the quality pass, not a queued Proposal. Each ledger row has a stable `issue_ref`, an issue family, and exactly one main route state. The exported `issue_ledger.json` is the full accounting record, while `deterministic_proposal_report.json` and `deterministic_proposal_report.md` summarize the funnel for Web/API review.

The route states separate successful automation from safety gates. `deterministic_covered` means a safe deterministic candidate became a queueable Proposal. `llm_residual` is the only leftover reasoning set shown to LLM subagents. `blocked_safety`, `blocked_apply`, and `blocked_evidence` are intentional gates for unsafe, unsupported, or ungrounded issues, not transport failures. `deferred_budget` means a valid issue was delayed by the deterministic family cap. Duplicate and stale routes prevent repeat work from prior accepted, rejected, or no-longer-current actions.

Latest funnel validation on `influenza_medical_v1`: `727` raw issues were fully accounted for with `0` unrouted issues. The report selected `15` deterministic proposals, dropped `13` candidate proposals, left `673` issues as `llm_residual`, and blocked `39` issues through existing decision memory. The queue's 15 proposals were all rejected by medical review, so `approval_queue.md` is now intentionally empty until the fingerprint-memory fix lands and a new run is generated.

Deterministic action-candidate generation is routed through the `lightrag/kb_iteration/deterministic_proposals/` family registry. The first high-yield generator batch covers entity cleanup, clinical modeling, diagnosis, treatment, risk/safety, and prevention. It emits only schema-valid candidates with explicit endpoints, required qualifiers, and existing apply support; TCM generation remains deferred until `medical_schema.py` defines explicit TCM predicates.

The proposal orchestrator also adds a role contract to each subagent task pack and prevalidates deterministic `action_candidates` before they are shown to the LLM. Unsafe candidates are kept in `rejected_action_candidates` for debugging instead of being copied into subagent prompts. Task packs carry `allowed_evidence_spans`, and `candidate_kg_expansion` output must reuse one exact evidence tuple from that allowlist. When Web/API runs set `skip_deterministic_subagent_calls=true` (the current Web default), task packs with safe complete candidates are marked `deterministic_only` and produce deterministic proposals without an LLM subagent call; blocked task packs are written to `subagent_outputs/index.json` with states such as `blocked_no_executable_candidate`, `blocked_unsafe_candidates`, or `blocked_no_grounding`.

The current Web/API defaults are `max_subagent_issues_per_task=4`, `max_subagent_proposals_per_task=2`, `strict_subagent_role_contracts=true`, `prevalidate_action_candidates=true`, `require_candidate_evidence_allowlist=true`, and `skip_deterministic_subagent_calls=true`. Role-specific Chinese subagent prompts live in `lightrag/kb_iteration/prompts/subagents/`.

Medical Relationship Schema v1 also includes explicit risk relations for hospital-facing KG use: `risk_factor_for`, `high_risk_for`, `increases_risk_of`, and `acute_exacerbation_of`. `risk_group_for` is deprecated in favor of `high_risk_for`. These relations prevent chronic disease, high-risk populations, acute exacerbation, outcomes, and severity states from being forced into `has_complication`.

After an approval pass is complete, keep `approval_queue.md` focused on proposals that still require a human decision. In the current cleaned workspace it is empty because the latest 15-proposal queue was fully reviewed and rejected. `improvement_backlog.md` now records the next engineering work rather than duplicating already-reviewed proposals. Long-term memory files such as `accepted_changes.md`, `rejected_changes.md`, `proposal_revision_requests.md`, and `agent_memory_summary.md` should be kept because later Agent runs use them to avoid repeating rejected proposals and already-applied fixes. Exact IDs alone are not enough; decision memory must also preserve semantic and execution fingerprints.

## 医学缺陷从检测到 Proposal 的漏斗

Web/API 对外仍使用 `agent_pipeline` 或 `loop` 两种 review mode；子 Agent 不是新的用户模式。当维护者选择 `agent_pipeline` 时，Proposal 阶段内部会经过 `Proposal Orchestrator + specialized subagents + merge gate`，把大量已检测缺陷压缩成少量可审阅 Proposal。

第一步始终是确定性检测器。它先扫描 KG，并把完整医学 Schema 缺陷表写入 `snapshots/quality_score.json` 的 `quality_score.details.medical_schema_issues`，把实体清理问题写入 `quality_score.details.entity_cleanup_issues`。这些记录只是“本轮检测到的问题”，不是待审批 Proposal，也不会自动进入 `approval_queue.md`。

Proposal Orchestrator 会先合并 `medical_schema_issues` 和 `entity_cleanup_issues`，再按医学问题家族、风险和证据把检测结果分组，生成有边界的任务包并写入 `proposal_task_packs.json`。任务包会保留确定性 issue ID、证据来源、风险线索和可执行 action candidate；如果缺陷很多，Orchestrator 只会按本轮预算挑选一部分问题送入子 Agent，而不是把所有 detected issues 都交给 LLM。现有确定性候选由 `deterministic_proposals/` 家族 registry 生成，第一批高产量规则覆盖实体清理、症状方向、诊断、治疗、风险安全和预防；只有 endpoint、类型、限定词和 apply 能力都明确时才生成候选。TCM 暂缓，等 `medical_schema.py` 明确 TCM 谓词后再做。

子 Agent 草稿必须先通过合并门校验，才可能进入审批：包括 JSON Schema 校验、证据 grounding 校验、`action_payload` grounding 校验、冲突检查、单轮 Proposal 数量限制，以及子 Agent 失败/超限处理。未通过的草稿会记录在 trace 或 merge report 中，但不会成为待审批项。`proposal_merge_report.md` 记录合并、丢弃和失败原因；`subagent_outputs/index.json` 是子 Agent 输出索引。

`approval_queue.md` 只包含本轮被选中、通过校验、并等待人工接受或拒绝的 Proposal。它不是全量缺陷清单，也不是 LLM 对 KG 的直接修改记录。常见计数含义如下：

- 检测问题数：本轮检测到的医学 KG 缺陷总数，对应 trace 字段 `raw_issue_count`。
- 送入子 Agent 问题数：实际送入子 Agent 的问题数，对应 trace 字段 `fitted_issue_count`。
- 待审批 Proposal 数：通过校验并等待人工接受/拒绝的 Proposal 数，对应 trace 字段 `selected_proposal_count`。
- 无效/丢弃数：因格式、证据、冲突、数量限制或子 Agent 失败而未进入审批的草稿数，对应 trace 字段 `invalid_subagent_output_count`、`dropped_proposal_count` 和 `dropped_proposal_reasons`。

LLM 不能直接修改 KG、prompt、规则、workspace 数据或 WebUI 行为。即使人工接受了 Proposal，真正写入 KG 的也只能是确定性 allowlisted apply engine；写入后必须重新生成 KG snapshot 和 `snapshots/quality_score.json` 作为变更证据。

## Medical Relationship Schema

When the review profile is medical, such as `clinical_guideline_zh` or `medical_kg`, the LLM agent prompts include Medical Relationship Schema v1 from `lightrag/kb_iteration/medical_schema.py`.

The schema is hospital-facing and separates:

- Knowledge-level facts from patient-instance facts.
- Medical concepts from terminology aliases.
- Diagnostic methods, specimens, observations, reports, and diagnostic conclusions.
- Drug indications, recommendations, dose regimens, contraindications, precautions, and adverse reactions.
- Vaccine or prevention product knowledge from recommendation rules and actual immunization events.
- Medical facts from guideline evidence and hospital organization/service routing.

Deterministic quality scoring also flags overloaded legacy relation keywords through `medical_schema_legacy_relation_count`. Examples include `推荐治疗`, `诊断依据`, `检测方法`, `适用于`, `预防措施`, `并发风险`, `剂量用法`, `病原分型`, and `指南建议`. These findings ask the LLM to generate reviewable proposals that migrate old edges toward canonical predicates such as `has_manifestation`, `has_diagnostic_criterion`, `criterion_requires`, `has_evidence`, `supports_or_refutes`, `has_indication`, `recommends`, `recommended_for`, `contraindicated_for`, `precaution_for`, `temporarily_deferred_for`, `has_dosing_regimen`, `targets_disease`, and `reduces_risk_of`.

Medical schema validation is shared by quality checks and proposal validation. `medical_schema.py` normalizes runtime entity types, understands lightweight clinical supertypes such as `Intervention`, `ClinicalFinding`, `Evidence`, and `MedicalConcept`, and validates relation instances through a single domain/range/qualifier contract. It also checks required qualifiers, allowed qualifiers, and constrained values such as `supports_or_refutes.polarity` and `recommended_for.purpose`. High-risk predicates such as `causative_agent`, `orders_test`, `has_result`, `recommended_for`, `contraindicated_for`, `precaution_for`, `temporarily_deferred_for`, `interaction_with`, `monitor_with`, `uses_specimen`, `belongs_to_drug_class`, and `evidenced_by` are no longer generic `MedicalConcept -> MedicalConcept` relations.

The schema does not make the agent an automatic clinical editor. It constrains explanation, evidence location, proposal generation, repair ranking, and judging. Any mutation to KG facts, prompts, relation rules, workspace data, or WebUI behavior still requires human approval.

Medical proposal validation also blocks several high-risk migration mistakes seen in hospital-facing KG review:

- `has_indication` must target a disease or clinical condition, not a bare pathogen such as `流感病毒`.
- `orders_test` must start from a disease, clinical condition, care process, or recommendation context, not from a bare pathogen such as `流感病毒`.
- `targets_disease` must target a disease or clinical condition such as `流行性感冒` or `季节性流感`, not a bare pathogen such as `流感病毒`.
- `has_manifestation` must target patient-observable symptoms, signs, or clinical findings; category labels such as `流感临床表现` and electrolyte/laboratory abnormalities such as `低钾血症` need different semantics or manual splitting.
- `causative_agent` must not model bacterial secondary-infection pathogens such as `肺炎链球菌` as the cause of viral influenza diseases such as `甲型流感`.
- `causative_agent` for typed influenza diseases such as `甲型流感` and `乙型流感` must not target generic `流感病毒`. Use or propose typed pathogen nodes such as `甲型流感病毒` and `乙型流感病毒`, then connect them upward with `is_a -> 流感病毒`.
- When a deterministic issue shows a legacy edge such as `流感病毒 -> 甲型流感` or `流感病毒 -> 乙型流感`, the Proposal Orchestrator should generate a `candidate_kg_expansion` that creates the typed pathogen node, adds `typed pathogen -> is_a -> 流感病毒`, adds `typed disease -> causative_agent -> typed pathogen`, and optionally retires the old generic edge after approval.
- `supports_or_refutes` should point from diagnostic evidence, tests, or findings toward the disease diagnosis, not from disease to a generic test.
- `supports_or_refutes` should not connect nonspecific labs or complication-assessment findings such as `血常规`, `血生化`, `动脉血气分析`, `丙氨酸氨基转移酶`, `天门冬氨酸氨基转移酶`, or `MRI` directly to `流行性感冒` diagnosis. Use clinical-finding, severity/complication assessment, or a more specific complication endpoint when grounded.
- `has_diagnostic_criterion` should be reserved for pathogen/etiology tests or true disease-defining criteria. Nonspecific labs or complication-assessment findings such as `血常规`, `血生化`, `动脉血气分析`, `丙氨酸氨基转移酶`, `天门冬氨酸氨基转移酶`, or `MRI` should use more precise evidence, observation, test-order, clinical-finding, or complication-specific semantics.
- In severe/critical influenza contexts, signs such as `鼻翼扇动`, `三凹征`, and `呼吸急促` are severity or diagnostic criteria. The deterministic candidate should prefer `流感重型 -> has_diagnostic_criterion -> 具体体征`, not downgrade them to generic `has_manifestation`.
- Nonspecific test edges should not be retired only because they are not direct influenza criteria. When the existing edge is a parent test panel such as `血常规`, `血生化`, or `动脉血气分析` pointing to a disease with a diagnostic-basis keyword, the Proposal Orchestrator can produce a deterministic `orders_test` action candidate such as `流行性感冒 -> orders_test -> 血常规`, with qualifiers preserving the diagnostic/severity-assessment context. LLM proposals that conflict with this deterministic action candidate are dropped in favor of the candidate.
- CT/MRI and other imaging tests must not be generalized to `流行性感冒 -> orders_test -> CT/MRI` when the evidence describes complication assessment. Attach CT/chest imaging to concrete pulmonary endpoints such as `流感肺炎` or `原发性病毒性肺炎`; attach MRI/CT in neurologic contexts to endpoints such as `急性坏死性脑病` or a specific neurologic complication.
- Bare lab analytes such as `肌酐`, `AST/ALT/LDH`, `肌酸激酶`, and `肌红蛋白` should not become direct `流行性感冒 -> orders_test -> 分析物` edges. Keep the parent panel as the ordered test and model analytes via `observes`, `has_result`, clinical findings, or grounded complication/severity assessment.
- Traditional Chinese Medicine syndrome indication edges should not stop at review-only proposals when the source edge is clear. The agent should propose an executable candidate that models the syndrome as a `ClinicalCondition` / TCM syndrome and preserves the treatment or drug `has_indication` relation.
- Grounded zanamivir split tasks must not stop at review-only proposals. For `扎那米韦->哮喘`, emit an executable split that models inhaled zanamivir as not recommended/precautionary for asthma or chronic respiratory disease and preserves route/risk qualifiers. For `扎那米韦->儿童`, emit an executable split with age, indication, route, dose, frequency, and duration qualifiers.
- `has_complication` must not model chronic underlying conditions such as `慢性阻塞性肺疾病(COPD)` as direct influenza complications. These usually belong to risk-factor/high-risk-population semantics unless the grounded endpoint is an acute exacerbation.
- Executable relation replacement proposals must copy the full current edge keyword string in `current_keywords`. If a legacy edge mixes multiple meanings, the Agent should split the relation or request review instead of silently dropping one meaning.
- Executable relation replacement proposals must not be no-ops. If source, target, predicate, and empty qualifiers are unchanged, the proposal should be skipped or converted into a request for clearer evidence.
- `is_a` must point from child/subtype to parent/supertype, for example `乙型流感病毒 -> is_a -> 流感病毒`; parent-to-subtype proposals such as `流感病毒 -> is_a -> 乙型流感病毒` are rejected.
- `recommended_for` proposals targeting broad populations such as `儿童` must include `purpose` (`treatment` or `prevention`) plus qualifying scope such as condition, age, route, context, risk, or reason. `contraindicated_for`, `precaution_for`, `not_recommended_for`, and `temporarily_deferred_for` should preserve their distinct safety meaning instead of collapsing into one generic population edge.

### Medical KG Schema Migration

The iteration Agent can detect medical relationship schema defects, ask the LLM to explain and propose repairs, queue human approval, and apply only allowlisted deterministic changes. LLM output is never treated as medical evidence or proof of mutation. The proof of mutation is the refreshed KG snapshot and `snapshots/quality_score.json`.

Current executable medical mutation proposals must include an `action_payload`. The deterministic apply layer currently supports safe subsets of:

- `medical_relation_schema_migration`, for verified relation replacement, direction migration, and tightly scoped relation retirement when an existing edge is an invalid or misleading medical fact that cannot be safely converted into a grounded canonical relation.
- `value_node_to_qualifier`, for conservative conversion of isolated value nodes into edge qualifiers.
- `candidate_kg_expansion`, for grounded candidate nodes/edges that still require human review before becoming KG facts. Its payload must include top-level `candidate_nodes` and `candidate_edges` arrays plus top-level `source_id`, `file_path`, `evidence_quote`, and `why_not_existing`. It may include `retire_edges` when the new candidates clearly replace a wrong existing edge. Nested evidence alone is insufficient because the merge gate must verify source grounding before approval.

The Agent context is budgeted separately from the full artifacts. `snapshots/quality_score.json` may keep the complete `medical_schema_issues` table for WebUI review, while LLM stage context receives compact, grounded samples plus total/visible/omitted summaries. This keeps large medical workspaces below the per-stage context gate without hiding the full audit data from the operator.

## Source Grounding

Treat `source_id`, `file_path`, and chunk linkage as the evidence trail for factual KG content. If evidence is missing, the correct response is to flag source repair, re-extraction, or manual review.

The agent can suggest navigation or category structure to organize already-extracted facts, but it must not invent clinical claims. A useful proposal explains the target, evidence, risk, expected metric change, and required approval.

## Approval Queue Behavior

Mutation proposals require explicit review. Examples include:

- `prompt_edit`
- `ontology_rule_change`
- `hierarchy_rule_change`
- `relation_rule_change`
- `workspace_rebuild`
- `kg_fact_correction`
- `web_display_change`

`proposals.py` enforces approval for known mutation proposal types and for unknown proposal types unless they are explicitly safe report notes. `write_approval_queue` writes only approval-required items, while `write_improvement_backlog` records all valid proposals.

Accepted changes should be copied into `accepted_changes.md` with the reviewer, reason, affected files, and verification needed. Rejected changes should be copied into `rejected_changes.md` with the reason and whether they may be reconsidered later. Rejected items become negative memory so future runs avoid repeating the same request.

## Diff And Follow-Up Runs

After an approved rebuild or rule change, generate a new snapshot and compare it with the previous one by using `compare_snapshots` and `write_diff_report`. A diff should be reviewed for added or removed nodes, relation keyword changes, entity type changes, quality deltas, and dangerous regression flags.

An iteration is accepted only when the evidence, quality metrics, and regression review support the change, or when the user explicitly accepts the trade-off.
