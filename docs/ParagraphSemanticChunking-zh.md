# 段落语义分块算法（Paragraph Semantic Chunking）

> 实现位置：[`lightrag/native_parser/docx/parse_document.py`](../lightrag/native_parser/docx/parse_document.py) 中 `extract_audit_blocks(..., fixlevel=None)` 默认路径。

## 1. 动机

传统的 RAG 文本分块方法（如固定字符 / token 滑窗、固定段数切分、按句子或换行切分）在面对真实业务文档（DOCX 报告、规范、白皮书等）时存在三类典型问题：

1. **跨主题串块**：滑窗/字符切分对内容语义无感知，常常把"上一个标题尾部的几句话"与"下一个标题开头"硬拼到同一个块里，导致检索器召回的上下文混入两个不相关的小节，污染答案语义。
2. **表格上下文丢失**：表格在 docx 中是独立结构，被裸切后失去其前置说明（"下表给出了……"）和后置解释，模型只看到孤立行列，难以正确解读；超大表格被拦腰截断后，下半段甚至失去表头信息。
3. **块粒度失衡**：切分要么过于细碎（一个短小节单独成块，token 不足以支撑完整推理），要么超出模型上下文（过长块直接被截断）。两者都会显著损伤检索召回质量与生成结果。

**段落语义分块** 的目标是利用 docx 自身携带的标题层级（outline level）、段落和表格结构信息，做一次"贴合作者本意"的切分：以标题语义为骨架、以表格完整性为约束、以平衡块大小为目标，输出一组 **顺序覆盖、互不重叠、首尾相接** 的文本块。

## 2. 设计原则

| 原则 | 含义 |
|---|---|
| 标题驱动 | 每一块以一个层级标题领起，块内全部内容都属于该标题之下；同级或更深层级标题作为内部结构保留，不再独立切分 |
| 不重叠、不丢失 | 任意两块的 paragraph 集合互斥，全部块按顺序拼接 == 原文（不做滑窗、不做交叠） |
| 表格优先完整 | 单个表格在 token 上限内绝不切分；必须切分时，与其前后正文上下文粘连，避免出现孤立的表格碎片 |
| 块大小受控 | 在 `IDEAL=6000 / MAX=8000` token 边界内做最均衡分配；过小块向同级或上级合并，过大块在合适锚点二次切分 |
| 层级感知合并 | 自底向上合并，浅层标题的块可以吸收紧随其后的深层标题块（"上位标题天然包住下位内容"） |

## 3. 关键阈值

| 常量 | 默认值 | 作用 |
|---|---|---|
| `IDEAL_BLOCK_CONTENT_TOKENS` | 6000 | 文本块理想大小（合并 / 切分目标） |
| `MAX_BLOCK_CONTENT_TOKENS` | 8000 | 文本块硬上限，超过即触发再切分 |
| `SMALL_TAIL_THRESHOLD` | 1000 | 文档尾部"零碎小块"整批吸收阈值（= (MAX-IDEAL)/2） |
| `MAX_ANCHOR_CANDIDATE_LENGTH` | 100 字符 | 长块再切分时被允许充当"锚点段落"的最大字符长度 |
| `TABLE_IDEAL_TOKENS` | 3000 | 表格切片理想大小 |
| `TABLE_MAX_TOKENS` | 5000 | 表格触发切分的硬上限（必须 < `IDEAL_BLOCK_CONTENT_TOKENS`） |
| `TABLE_MIN_LAST_CHUNK_TOKENS` | 1600 | 表格末片最小阈值（小于则回吞至前一片） |

阈值之间的不变式：`TABLE_MAX_TOKENS < IDEAL_BLOCK_CONTENT_TOKENS < MAX_BLOCK_CONTENT_TOKENS`，保证一个完整表格片永远可以容纳进一个文本块。

## 4. 算法工作原理

整体分四个阶段，对 docx body 做一次顺序遍历后，对结果做"长块再切分 + 小块合并"两轮整形：

```
docx body
   │
   ▼
[阶段 A] 标题驱动的初始线性切分 ────────────► 初始块序列
   │      （遇到任意级别标题就启动新块）
   │
   ▼
[阶段 B] 表格的预切分与上下文粘连 ──────────► 含表格切片的初始块
   │      （首片粘前文 / 中间片独立 / 末片粘后文）
   │
   ▼
[阶段 C] split_long_block：锚点驱动的再切分
   │      （超 8000 token 的块按短段落切开）
   │
   ▼
[阶段 D] merge_small_blocks：自底向上的层级感知合并
          （同级合并 → 跨级吸收 → 尾部整批吸收）
   │
   ▼
最终块列表（顺序、不重叠、覆盖全文）
```

### 4.1 阶段 A：标题驱动的初始线性切分

逐 body 元素扫描，维护若干"当前块"状态：累积段落 `current_paragraphs`、当前 heading、heading 栈、`parent_headings`、`has_body_content` 等。

- **标题判定**：通过 docx 的 outline level（0–8 → 内部 level 1–9）识别；段落语义分块模式下，**任意级别的标题都是切分点**。
- **遇到标题**：
  1. 若上一个块已经有正文（`has_body_content == True`），通过 `_flush_current_block` 把它送入阶段 C；
  2. 把当前标题作为新块的第一段，并维护 heading 栈（同级覆盖、深层裁剪）；
  3. `parent_headings` 由 heading 栈中所有 **严格更浅** 的层级按层级升序排出。
- **关键不变量**：连续多个标题不会产生空块——只有出现正文之后再遇到标题，才会真正落盘上一块。这等价于："以某一层级的标题开始，后续内容必然属于该标题之下；遇到下一个标题（任意级别）即结束当前块"。
- **普通段落 / 表格**：直接 append 到 `current_paragraphs`，并标记 `has_body_content = True`。

文档遍历完成后，对剩余 `current_paragraphs` 调用一次 `_flush_current_block`，落最后一块。

### 4.2 阶段 B：表格预切分与上下文粘连

表格内容会被 `<table>...</table>` 标记后参与块构造。当表格 token 数超过 `TABLE_MAX_TOKENS (5000)` 时，按行边界切片：

- 目标片数 `= max(ceil(total / TABLE_IDEAL_TOKENS), ceil(total / TABLE_MAX_TOKENS))`；
- 末片若小于 `TABLE_MIN_LAST_CHUNK_TOKENS (1600)`，则回吞至前一片（前提合并后仍 ≤ `TABLE_MAX_TOKENS`）。

切完后每个表格切片携带 `table_chunk_role`，并按角色与正文粘连：

| 角色 | 粘连策略 | 目的 |
|---|---|---|
| **first** | 直接 append 到当前块尾，与表格 **前文** 合成同一块 | 表格前文（"下表说明……"）不被孤立 |
| **middle** | 立即落上一块，自身作为独立块输出（heading 加 `[表格片段N]` 后缀，并继承 `table_header`） | 中间整片单独成块，但保留表头 |
| **last** | 同样先落上一块，再 append 到新累积的开头，等待与表格 **后文** 合块 | 表格的后文解释不被孤立 |

由此得到核心保证：**表格前文 + 表格首片 = 同一块；表格末片 + 表格后文 = 同一块；只有完全居中的中间片才作为独立块**。这是"避免孤立表格内容"诉求的直接落实。

### 4.3 阶段 C：`split_long_block` —— 锚点驱动的再切分

`_flush_current_block` 把累积块送入 `split_long_block`，对超过 `MAX_BLOCK_CONTENT_TOKENS (8000)` 的块进行均衡再切分：

1. **目标块数** `target_blocks = max(ceil(total / IDEAL), ceil(total / MAX))`，平均大小 `target_size = total / target_blocks`；
2. **筛选锚点候选**：遍历段落，挑选 **非表格、字符长度 ≤ 100** 的短段落（如"概述""背景""详细说明"），并记录其在文档中的累计 token 偏移。**表格永远不可作锚点**，从而切点天然落在文本段落上，避免劈开表格；
3. **均衡选锚**：对 `i = 1..target_blocks-1`，理想切点 `ideal_position = i * target_size`，从候选中选 `|position - ideal|` 最小者；选完按文档顺序排序；
4. **构块**：每个锚点段落 **作为下一个子块的第一段**（晋升为新 heading），其前的段落组成一块；首块沿用原 heading，后续子块的 `parent_headings = parent + [原 heading]`；
5. **后验校验**：任何子块仍 > MAX 时，对其保留的段落集合递归再切；若全无候选锚点，则报错并退出（提示用户在文档中插入短段落作为切分点）；
6. 若该块由"表格末片粘连而成"，则用末片携带的 `_chunk_heading / _table_header` 元数据替换 heading 与表头。

### 4.4 阶段 D：`merge_small_blocks` —— 自底向上的层级感知合并

对全部初始块进行整形，按 **从最深 level 到最浅 level** 处理；每层走两个相位：

#### Phase A：同级合并

对 `tokens < IDEAL` 的"未饱和"块，依次尝试 forward / backward 合并相邻的 **同级** 块。合并条件：

- 合并后 token ≤ `MAX_BLOCK_CONTENT_TOKENS`；
- 满足 `table_chunk_role` 约束（见下表）；
- backward 合并时前块也需 `< IDEAL`；

合并后 heading / parent_headings / level 取保留方；`uuid` 取首块、`uuid_end` 取末块；`table_header` 优先继承已存在者；融合后 `table_chunk_role` 重置为 `none`。一个块达到 IDEAL 即被锁定，不再参与本相位。

**尾部整批吸收**：当一个 ≥ IDEAL 的块后面紧跟一串同级小块、且这些小块 token 总和 < `SMALL_TAIL_THRESHOLD (1000)`、且合并总量 ≤ MAX，则一次性把它们全部并入前块——避免文档末尾留下零碎尾巴；遇到 `middle` 表格切片立即停止吸收。

#### Phase B：跨级吸收

允许 **较浅层级（level 数小）的未饱和块吸收紧邻的更深层级块**：

- forward：current 是浅层 → 吸收随后的 deeper 块；
- backward：current 是深层 → 让紧邻的 shallower 前块吸收自己；

合并后保留较浅一层的 heading / parent_headings / level——这正契合"上位标题天然包住下位内容"的语义。

#### 表格角色对合并方向的约束

| `table_chunk_role` | Phase A 前并 | Phase A 后并 | Phase B 前并 | Phase B 后并 |
|---|---|---|---|---|
| `none` | ✓ | ✓ | ✓ | ✓ |
| `first` | ✓（向后吞内容） | ✗ | ✓ | ✗ |
| `last` | ✗ | ✓（向前吞内容） | ✓ | ✓ |
| `middle` | ✗ | ✗ | ✗ | ✗ |

这套约束保证：切开的表格首片只能继续"吃"后面内容，末片只能继续"吃"前面内容，中间片绝不再被裹挟——孤立的中间表格片段不会与不相关内容错配。

## 5. 输出结构

每个最终块是一个 dict，关键字段：

| 字段 | 说明 |
|---|---|
| `heading` | 该块的领起标题（可能是文档原始标题、锚点晋升标题、或表格片段标题 `… [表格片段N]`） |
| `parent_headings` | 所有更浅层级的祖先标题，按层级升序 |
| `level` | 领起标题的层级（1 = Heading 1） |
| `content` | 拼接后的文本（含 `<table>...</table>` 标记） |
| `uuid` / `uuid_end` | 块首段 / 块尾段（或表格末单元格末段）的 docx `w14:paraId` |
| `type` | 固定为 `"text"` |
| `table_chunk_role` | `none / first / middle / last`，用于下游识别表格切片角色 |
| `table_header`（可选） | 当块包含被切的表格时，保留原表头行，便于下游补回上下文 |

## 6. 整体不变量

- **顺序覆盖、互不重叠**：所有阶段都是"切前一段累积 + 启动下一段累积"，无窗口、无重叠。按顺序拼接所有块的 `content` 即可还原原文。
- **以标题领起**：每个块的 heading 来自三种来源：文档原始标题（阶段 A）、锚点段落晋升的"伪标题"（阶段 C）、或合并后保留的较浅级标题（阶段 D Phase B）；同级内容会被合并以避免过碎（阶段 D Phase A）。
- **表格保护**：`TABLE_MAX_TOKENS` 以下不切表；切了之后通过 `is_first / last` 与正文粘连、`middle` 独立成块；后续锚点切分明确排除 `is_table` 候选；合并阶段通过 `table_chunk_role` 约束方向。
- **大小约束**：除递归切分仍找不到锚点的极端情形会显式报错外，所有最终块满足 `tokens ≤ MAX_BLOCK_CONTENT_TOKENS`；合并器倾向把 `< IDEAL` 的块拉到 IDEAL 附近，但不越过硬上限。

## 7. 与传统分块方法的对比

| 维度 | 固定字符/Token 滑窗 | 单纯按标题切（不合并） | **段落语义分块** |
|---|---|---|---|
| 跨主题串块 | 高频（窗口跨标题） | 不会 | 不会（标题严格领起） |
| 表格完整性 | 经常拦腰切 | 大表会失控 | 小表整体保留；大表按行切并保留表头 |
| 表格上下文 | 经常丢失 | 经常孤立 | 首/末片与正文粘连，中间片显式标注 |
| 块大小均衡 | 依赖固定窗口，无法适配标题密度 | 完全失衡（短小节 / 长章节并存） | IDEAL/MAX 双阈值 + 锚点再切分 + 小块合并 |
| 块边界可解释 | 弱（任意位置） | 强（标题边界） | 强（标题边界 + 短段落锚点） |
| 检索召回质量 | 上下文混杂、表格碎片化 | 粒度严重不均 | 粒度受控、语义内聚、表格完整 |

## 8. 特点优势

1. **语义内聚**：每个块只承载一个标题之下的内容，避免上一节尾部混入下一节开头，显著降低检索召回噪声。
2. **表格友好**：从"是否切"、"怎么切"、"切了之后怎么粘"三层保护，消除孤立表格内容；中间切片附带 `[表格片段N]` 标注与表头，下游 LLM 可以正确还原结构。
3. **粒度可控**：以 `IDEAL=6000 / MAX=8000` 为目标做双向调整——过小者向同级或上级合并、过大者沿短段落锚点均衡再切，避免传统方法常见的"一边过碎、一边超长"。
4. **层级感知**：合并器按 level 自底向上推进，浅层标题块吸收深层小块，保留作者构思的层级语义；尾部小块整批吸收策略消除文档结尾的零碎残片。
5. **可追溯**：每个块保留 `uuid / uuid_end / level / parent_headings / table_chunk_role`，便于在审计、定位、可视化等场景把块映射回 docx 中的具体段落范围。
6. **失败可见**：当块无可用锚点导致无法切分到 MAX 以下时，函数显式报错并给出可执行的修复建议（在源文档中插入短段落作为切点），不会静默产出超长块。

## 9. 适用场景

- **适用**：结构化办公文档（DOCX 报告、规范、白皮书、合同、审计材料、技术文档），尤其是含丰富标题层级与表格的内容。
- **不适用**：
  - 几乎不带标题层级的纯散文 / 聊天记录 / 邮件正文 —— 标题驱动失去抓手，应回退到滑窗或句子级切分；
  - 极端密集的小标题文档（每段都是 Heading），此时块会非常多，需依赖阶段 D 的合并器，并可注意 `IDEAL_BLOCK_CONTENT_TOKENS` 是否合适；
  - 非 docx 格式（PDF / HTML / Markdown）—— 该实现强依赖 docx 的 outline level 与 paraId，需要为其他格式实现等效的标题与段落抽取层。

## 10. 调试与可观测

- 通过 `extract_audit_blocks(file_path, debug=True)` 开启调试输出，会打印阶段 C 切分前后的 token 分布、阶段 D 每轮 / 每相位的合并次数、最终块数与超大块预警等。
- `parse_warnings` 字典在解析过程中累积非致命告警（例如缺失 `w14:paraId` 的段落或表格数），调用方（adapter / CLI）可据此输出一行汇总警告。
- 可以通过传入 `fixlevel=N` 切到"固定层级模式"（仅按 `level ≤ N` 的标题切分、关闭再切分与合并），用于对比、调参或紧急规避。
